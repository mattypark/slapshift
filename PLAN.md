## Context

Build a macOS menu-bar app that detects when you physically slap your MacBook (via the internal accelerometer on Apple Silicon), counts the number of slaps within a debounce window (1 / 2 / 3), and triggers a configured "mode" — a bundle of apps to open, apps to quit, URLs to launch, and (optionally) macOS Focus mode to enter via Shortcuts.

Inspired by [slapmac.com](https://slapmac.com) — same hardware trick, productivity framing instead of comedy. Personal use first, then ship as a notarized DMG with a Stripe-powered subscription. Target: $20K month-one revenue (stretch), $5K conservative.

## Problem Statement

Every morning Matthew opens 5-8 apps by hand (VSCode, Cursor, Chrome with specific tabs, Spotify, Terminal). It's a 30-second ritual repeated daily. Same problem for school applications — opening 10 college portals and a Common App tab in sequence. The action is mechanical, deterministic, and ripe for one-gesture replacement. SlapMac proved the slap-detection hardware path works on Apple Silicon. Nobody has shipped a productivity tool on top of it.

## What Makes This Cool

The gesture itself is the marketing. "I slap my MacBook and my whole workspace transforms" is a 6-second Twitter video that explains the product completely. The novelty hook is high (people stop scrolling), the demo is concrete (real apps open, real work begins), and the price point is impulse-buy. There is currently zero serious competition — SlapMac exists but is a sound-effects toy.

The technical lift is nearly identical. Product framing changes from $9 toy to $15-19 productivity tool.

## 3.3 IOKit AppleHIDMotion — Detailed Plan

### What we're betting on

Apple Silicon MacBooks have an internal motion sensor exposed via IOKit as a HID device under `AppleHIDMotionSensor` (the exact service name has varied across macOS versions). SlapMac reads it; we'll do the same. Polling at 100Hz gives 10ms resolution — enough to catch a slap impulse (typically a 30-80ms acceleration spike).

### Weekend 1 spike (CLI, no UI, throwaway)

A standalone Swift Package at `spike/SlapSpike/` that:

1. Calls `IOServiceMatching("AppleHIDMotionSensor")` (try variants if first fails: `AppleHIDMotion`, `IOPlatformHIDMotion`, `AppleAccelerometerHIDMotion`)
2. Opens the service via `IOServiceOpen`
3. Polls `IOHIDDeviceGetReportWithCallback` or `IOServiceCallWithStruct` at 100Hz
4. Prints `(x, y, z, magnitude)` for 10 seconds
5. Bonus: highlights any sample where `magnitude > 1.5g`

**Pass criteria:** non-zero, non-constant accelerometer values when Matthew taps the MacBook. Clear spike when Matthew slaps it.

**If it fails:**
- Try alternative service names
- Try `CMMotionManager` (long shot — it's iOS-only, but worth confirming)
- Pivot: dig into how SlapMac specifically reads the sensor (their app is on disk after install — IOKit calls are visible via `dtruss`)
- If all fail, STOP. Don't build the product on a broken foundation.

The spike is throwaway. The real `MotionPoller.swift` in the app will be ported from it after the API is confirmed.

### Slap classification algorithm

```
                  ┌─ idle: magnitude < threshold for >300ms
                  │
   sample(g) ─────┼─ peak detected: magnitude > threshold (default 1.4g)
                  │     │
                  │     ├─ record peak timestamp
                  │     │
                  │     ├─ start 400ms debounce window
                  │     │
                  │     ├─ during window: count additional peaks
                  │     │   (require > 80ms gap between peaks to avoid ringing)
                  │     │
                  │     └─ window expires:
                  │            n = number of peaks observed
                  │            if n in {1,2,3}: emit SlapEvent(count: n)
                  │            else: discard (likely a drop or shake)
```

Implementation in `SlapClassifier.swift`. Threshold is user-tunable via SensitivityView. Test coverage: feed synthetic sample streams (recorded from spike) and assert the classifier returns expected counts.

**Latency budget:**
- Single slap: 400ms (debounce window must expire to confirm "only one")
- Two slaps: same 400ms (window resets on second peak — no, actually that's a problem; window resets means worst case is 800ms)
  - **Fix:** debounce window is from FIRST peak, not last. So 400ms total regardless of count.
- Menu bar flash fires on FIRST peak (immediate feedback, even before classification)

### Edge cases handled

- **Sleep/wake:** stop polling when system sleeps (`NSWorkspace.willSleepNotification`), restart on wake. Drain any stale samples.
- **Lid closed:** if external display attached and lid closed, accelerometer values may be noisy. Pause detection when `IOPMAssertion` reports clamshell. (Punt to v1.1 if it complicates v1.)
- **Multi-monitor:** irrelevant — actions are screen-agnostic.
- **Action in flight:** if a mode is currently executing (apps still opening), ignore new slaps for 2 seconds. Prevents double-firing during the choreographed app launch.
- **App not installed:** if a mode references an app that's been uninstalled, log + skip silently. Don't crash.

## 3.4 Mode Persistence

Stored as JSON at `~/Library/Application Support/SlapShift/modes.json`. Why not UserDefaults: easier to inspect/edit/backup, easier to support import/export later, doesn't pollute the plist.

```json
{
  "version": 1,
  "modes": {
    "1": {
      "name": "Coding",
      "icon": "code",
      "openApps": ["com.microsoft.VSCode", "com.google.Chrome"],
      "quitApps": ["com.tinyspeck.slackmacgap", "com.hnc.Discord"],
      "openURLs": ["http://localhost:3000"],
      "focusShortcut": "Coding Focus",
      "enabled": true
    },
    "2": { "name": "Apply", "...": "..." },
    "3": { "name": "Wind Down", "...": "..." }
  }
}
```

App identifiers are bundle IDs (resolved via `NSWorkspace.urlForApplication(withBundleIdentifier:)`), not paths — survives /Applications reorgs.

Atomic writes via `Data.write(to:options: .atomic)`. Read once at launch, hold in memory, write on user edits.

## 3.5 Action Executor — Orchestration

When `ModeRouter` receives `SlapEvent(count: 1)`:

1. Look up `modes["1"]`. If disabled or missing, no-op.
2. Flash menu bar icon green for 200ms.
3. Run quits first (sequential, `NSRunningApplication.terminate()` then `forceTerminate()` after 2s if still alive).
4. Run app opens in parallel (`NSWorkspace.openApplication(at:configuration:)` for each).
5. Run URL opens in parallel (`NSWorkspace.open(_:)`).
6. If `focusShortcut` set: `Process()` running `/usr/bin/shortcuts run "Coding Focus"`. Don't wait — fire and forget.

All steps logged to `~/Library/Logs/SlapShift/actions.log` (rotating, 5 files x 1MB). Used for debugging customer reports.


### License key format

`SS-XXXX-XXXX-XXXX-XXXX` where X is base32-alphabet character (no I/L/O/0/1 to avoid OCR confusion). Generated server-side as `SS-` + 16 chars from `crypto.randomBytes(10)` base32-encoded. Stored in Supabase as primary key.

### Validate endpoint contract

```
POST /api/validate
Body: { key: "SS-XXXX-...", machineId: "<hashed serial>" }
Response 200: { active: true, expiresAt: "2026-06-19T00:00:00Z", email: "..." }
Response 200: { active: false, reason: "canceled" | "unpaid" | "not_found" }
Response 401/500: error states
```

Machine ID is `sha256(IOPlatformExpertDevice.IOPlatformSerialNumber + license_key)`. Lets us soft-cap concurrent installs without storing raw serial numbers.

### Cancellation flow

Stripe webhook `customer.subscription.deleted` → mark row `active = false`. Next app launch validates → 200 `{active: false}` → paywall.

### Offline tolerance

Cache last successful validation in `~/Library/Application Support/SlapShift/license.json`:
```json
{ "key": "SS-XXXX-...", "lastValidated": "2026-05-19T12:00:00Z", "active": true }
```

If app launches offline AND cache is fresh (<30 days), proceed. >30 days, show paywall with "retry when online" CTA. This is a soft grace period for travel/Wi-Fi issues — not a piracy vector because the cancellation eventually propagates.

### Why not check on every action

Considered: validate every 24 hours via a background timer. Rejected because:
1. User pays monthly. Daily validation is overkill.
2. Cancellation propagates within 30 days max via cache expiry. Acceptable.
3. Fewer moving parts. Fewer crash modes. Validating only at launch boundaries means no concurrent state issues.

## 3.7 Focus Mode Integration

**Decision (locked):** ship apps+URLs+quits as core (always works), Focus integration as bonus layer via first-run installer.

### First-run installer flow

On first app launch (after license validates):
1. `OnboardingView` shows a "Set up Focus modes (optional)" step.
2. Three buttons: "Install Coding Focus", "Install Apply Focus", "Install Wind Down Focus".
3. Each button opens a bundled `.shortcut` file via `NSWorkspace.open(_:)`. macOS handles the import dialog.
4. User clicks "Add Shortcut" in the system dialog.
5. After each install, the corresponding mode card in settings shows a green check.

If user skips: modes still fire (apps open/quit, URLs launch), Focus step is just a no-op. `FocusRunner` checks `shortcuts list` output and silently skips if the shortcut isn't installed.

### Bundled shortcuts

Create three `.shortcut` files in Apple's Shortcuts.app, export them, bundle in `Resources/DefaultShortcuts/`:

- **CodingFocus.shortcut** — Turn on "Do Not Disturb" focus
- **ApplyFocus.shortcut** — Turn on "Personal" focus
- **WindDownFocus.shortcut** — Turn on "Sleep" focus

These are simple "Set Focus" shortcuts. Users can edit them after install if they want different behavior.

## 3.8 Build & Release Pipeline

### Local dev loop

1. Open `app/SlapShift.xcodeproj` in Xcode.
2. ⌘R to run. Debug build runs as a regular menu bar app.

### Release build (manual for v1)

`ops/release.sh`:
```bash
#!/usr/bin/env bash
set -euo pipefail
VERSION="${1:?usage: release.sh <version>}"

# 1. Bump version in Info.plist
agvtool new-marketing-version "$VERSION"

# 2. Archive
xcodebuild -project app/SlapShift.xcodeproj \
  -scheme SlapShift \
  -configuration Release \
  -archivePath build/SlapShift.xcarchive \
  clean archive

# 3. Export signed .app
xcodebuild -exportArchive \
  -archivePath build/SlapShift.xcarchive \
  -exportPath build/export \
  -exportOptionsPlist ops/exportOptions.plist

# 4. Create DMG
ops/create-dmg.sh build/export/SlapShift.app build/SlapShift-$VERSION.dmg

# 5. Notarize
xcrun notarytool submit build/SlapShift-$VERSION.dmg \
  --keychain-profile "slapshift-notary" \
  --wait

# 6. Staple
xcrun stapler staple build/SlapShift-$VERSION.dmg

echo "Done: build/SlapShift-$VERSION.dmg"
```

Manual notarization for first 3 releases. After that, automate via GitHub Actions if shipping speed matters.

## 3.10 Test Strategy

### What gets tested (XCTest)

**`SlapClassifierTests.swift`** — ★★★ critical
- Synthetic sample streams → expected slap count
- Edge: peak at exactly threshold boundary
- Edge: 4 peaks within window (should discard as noise)
- Edge: peak followed by 350ms gap then another peak (within window? boundary case)
- Edge: zeros-only stream → no events
- Edge: sustained vibration > threshold → counted as one slap, then idle re-entry

**`ModeStoreTests.swift`** — ★★ behavior
- Load missing file → returns defaults
- Save then load → round-trips
- Corrupt JSON → falls back to defaults, logs warning
- Concurrent reads while write in progress → no crash (use atomic write)

**`LicenseValidatorTests.swift`** — ★★ behavior + error paths
- Mock 200 active → returns active
- Mock 200 inactive → returns inactive
- Mock network error + fresh cache → returns cached active
- Mock network error + stale cache → returns paywall

**`MotionPollerTests.swift`** — ★ smoke
- Init doesn't crash
- Polling can be stopped
- Real hardware tests require a Mac, run manually

### What does NOT get tested

- SwiftUI views (low value, brittle, view structure changes constantly)
- NSWorkspace integration (integration test on a real Mac, manual)
- The shortcuts CLI integration (manual: install shortcut, run mode, verify Focus)

## R.2 P0/P1 Issues Found in Current Code

### P0 — Focus shortcuts will silently fail (confidence: 10/10)

`ActionExecutor.swift` calls `/usr/bin/shortcuts run "SlapShift: Set Focus to \(name)"` but no `.shortcut` files exist on disk yet. Every Coding/Apply/WindDown activation will silently no-op the Focus step. Demo failure mode: "the slap worked, the apps opened, but nothing changed about my notifications" — kills the viral demo.

**Fix:** Build three `.shortcut` files in Shortcuts.app named exactly:
- `SlapShift: Set Focus to Do Not Disturb`
- `SlapShift: Set Focus to Personal`
- `SlapShift: Set Focus to Sleep`

Each is a single "Set Focus" action. Export, bundle in `Resources/DefaultShortcuts/`. Onboarding installs them via `NSWorkspace.open(_:)` of each bundled file (macOS handles the "Add Shortcut" dialog). Phase A blocker.

### P1 — Apps open in background (confidence: 9/10)

`ActionExecutor.swift:32` sets `NSWorkspace.OpenConfiguration.activates = false`. Result: VSCode launches but doesn't take focus. The user has to ⌘-Tab to it. Kills the "slap → instant workspace" feel.

**Fix:** Set `activates = true` for the FIRST app in `appsToOpen`, `false` for the rest (so the user doesn't get focus-stolen mid-typing if they typo-slap). Add a unit test that mocks the config and asserts activation policy per-app.

### P1 — No Input Monitoring permission UX (confidence: 9/10)

IOKit HID access requires Input Monitoring permission on macOS 13+. Without it, `MotionPoller` opens the device but every read returns empty. There's no error path that tells the user "go grant the permission." On first launch the app will appear broken.

**Fix:** Onboarding step that:
1. Detects denial via attempting a single read and checking sample count after 500ms
2. Deep-links to System Settings: `NSWorkspace.shared.open(URL(string: "x-apple.systempreferences:com.apple.preference.security?Privacy_ListenEvent")!)`
3. Shows a screenshot of the toggle the user needs to flip
4. Polls in background, advances onboarding step when sensor goes live

### P2 — Code quality nits

- `SlapClassifier.swift:87` — `scheduleEmit(at deadline:)` ignores `deadline` parameter and uses `.now() + windowSeconds`. Dead parameter. Either honor it or remove it.
- `MotionPoller.swift` magnitude filter `0.3 < mag < 50.0` is a magic number. Add a `// MAGIC` comment explaining the sanity bounds (sensor occasionally returns garbage on wake — verified during calibration).
- `ModeStore.save()` swallows errors via `print`. Settings UI has no way to surface "save failed." Fix: publish `lastSaveError: Error?` on the store, show a red banner in SettingsView header.
- `DefaultModes.swift` hardcodes bundle IDs (incl. `com.todesktop.230313mzl4w4u92` for Cursor) that may not be installed on a buyer's Mac. Mitigation: `ActionExecutor` already skips non-installed apps silently (verified §3.5). Acceptable, but onboarding should preview which default-mode apps are missing and let the user remove them in one click.

### What this changes vs §3.6

### Deep link plumbing

- `Info.plist` declares URL scheme `slapshift` (`CFBundleURLTypes` → `CFBundleURLSchemes = ["slapshift"]`).
- `AppDelegate.applicationWillFinishLaunching` registers handler via `NSAppleEventManager.shared().setEventHandler(_:andSelector:forEventClass:andEventID:)` with `kInternetEventClass`/`kAEGetURL`.
- Handler parses `slapshift://license/{token}` → writes `LicenseCache` → posts `Notification.Name.licenseActivated` → `LicenseGate` dismisses paywall sheet.
- Web `success` page reads `session_id` query param → fetches token from `/api/license-by-session?session_id=…` → `window.location.href = "slapshift://license/" + token`.

## R.6 Build & Release Pipeline (P0, replaces §3.8)

Single script `ops/release.sh`:

```bash
#!/usr/bin/env bash
set -euo pipefail
VERSION="${1:?usage: release.sh <version>}"
ROOT="$(cd "$(dirname "$0")/.." && pwd)"
cd "$ROOT"

# 1. Archive
xcodebuild -project app/SlapShift.xcodeproj \
  -scheme SlapShift -configuration Release \
  -archivePath build/SlapShift.xcarchive \
  MARKETING_VERSION="$VERSION" \
  clean archive

# 2. Export signed .app (Developer ID, hardened runtime)
xcodebuild -exportArchive \
  -archivePath build/SlapShift.xcarchive \
  -exportPath build/export \
  -exportOptionsPlist ops/exportOptions.plist

# 3. Create DMG
ops/create-dmg.sh build/export/SlapShift.app build/SlapShift-$VERSION.dmg

# 4. Notarize (blocking — `--wait` polls Apple until done, ~3-8 min)
xcrun notarytool submit build/SlapShift-$VERSION.dmg \
  --keychain-profile "slapshift-notary" --wait

# 5. Staple
xcrun stapler staple build/SlapShift-$VERSION.dmg

# 6. Verify
spctl -a -t open --context context:primary-signature -v build/SlapShift-$VERSION.dmg
echo "Done: build/SlapShift-$VERSION.dmg"
```

`ops/exportOptions.plist`:
```xml
<dict>
  <key>method</key><string>developer-id</string>
  <key>signingStyle</key><string>automatic</string>
  <key>teamID</key><string>YOUR_TEAM_ID</string>
  <key>destination</key><string>export</string>
</dict>
```

**Notary credential setup (one-time, manual, before first run):**
```
xcrun notarytool store-credentials "slapshift-notary" \
  --apple-id "<your apple id>" \
  --team-id "<TEAM_ID>" \
  --password "<app-specific password from appleid.apple.com>"
```

Per CLAUDE.md security rules: I will not read your TEAM_ID or App-Specific Password. You run this command yourself once. After that the keychain holds it; the release script just references the profile name.

**Phase A dry-run procedure:** before notarizing real SlapShift, build a 1-file "hello world" macOS app, run the full pipeline against it. If notarization succeeds + `spctl` clears it, the pipeline is proven. Then swap in SlapShift.

## R.8 Worktree Parallelization

Phase A is strictly sequential (each step depends on the previous build artifact).

Phase B has one parallelization opportunity:
- **Lane A:** Web API routes + Supabase schema + success page (`web/app/api/**`, `web/app/success/page.tsx`)
- **Lane B:** Swift License/* + PaywallView + URL scheme handler (`app/SlapShift/License/**`, `app/SlapShift/UI/PaywallView.swift`)

These touch disjoint module directories — `web/` vs `app/`. Safe to develop in parallel worktrees. Merge gate is the contract on `/api/validate` and `/api/checkout` request/response shapes. Lock the contracts (R.3 above) before splitting lanes.
